## 표준 내부 스키마

정산상세 원형은 플랫폼별로 다르지만, import layer가 내보낼 표준 필드는 다음으로 고정한다.

| 표준 필드 | 의미 |
| --- | --- |
| `source_file` | 원본 파일명 |
| `source_sheet` | 원본 시트명 |
| `source_row_no` | 원본 행 번호 |
| `platform` | 상위 폴더 기준 플랫폼/판매처 |
| `settlement_month` | 파일명/폴더/시트에서 추출한 정산월 |
| `raw_title` | 작품명/상품명/콘텐츠명 원문 |
| `external_content_id` | 원본 정산서에 있는 외부 ID/작품코드/상품ID |
| `sale_amount` | S2 전송자료의 `판매금액` 후보 |
| `settlement_base_amount` | S2 전송자료의 `정산기준액` 후보 |
| `offset_amount` | S2 전송자료의 `상계금액` 후보. 없으면 원칙적으로 0 후보 |
| `schema_adapter` | 감지된 adapter 이름 |
| `schema_status` | `parsed`, `needs_manual_review`, `schema_error` |

주의: 원본의 `정산금액`, `정산액`, `공급가액`, `입금액`이 모두 같은 의미는 아니다. 이 문서는 헤더 기반 스키마 확정이고, 세전/세후/수수료 차감 의미는 adapter 구현 전 플랫폼별 확인이 필요하다.

## Adapter 계약

각 adapter는 다음 인터페이스를 만족해야 한다.

```text
detect(file_meta, workbook_meta, header_signature) -> confidence
parse(workbook, sheet) -> StandardSettlementRows
validate(rows) -> ValidationReport
```

`detect`는 플랫폼 폴더명만 믿지 않고 시트명, 헤더 행, 헤더 시그니처를 함께 본다.

`parse`는 원본 행 번호를 보존한다. 그래야 나중에 S2 전송자료에서 문제가 생겼을 때 원본 정산상세 행으로 돌아갈 수 있다.

`validate`는 다음을 반드시 검사한다.

- 제목/ID 키가 비어 있는 행
- 판매금액/정산기준액이 숫자로 파싱되지 않는 행
- 합계행/소계행/주석행이 섞인 행
- 음수/취소/상계 행
- 구매/대여/정액권/면세/과세가 분리된 행

## S2 전송자료 연결

S2 전송자료는 최종적으로 다음 4컬럼만 필요하다.

| S2 컬럼 | 표준 스키마 입력 |
| --- | --- |
| `판매채널콘텐츠ID` | KISS/S2 판매채널콘텐츠 목록과 매칭한 `schnCtnsId` |
| `판매금액` | `sale_amount` |
| `정산기준액` | `settlement_base_amount` |
| `상계금액` | `offset_amount`, 없으면 0 후보 |

중요한 점은 원본 정산상세에 `판매채널콘텐츠ID`가 없는 경우가 많다는 것이다. 따라서 원본 adapter는 S2 ID를 직접 만들지 않고, `raw_title`과 `external_content_id`를 최대한 보존해 KISS/S2 후보 목록과 매칭하게 해야 한다.
