## 읽는 법

아래 표의 표준 필드는 매핑 시스템에 먹일 표준 DataFrame 기준이다.

| 표준 필드 | 의미 |
| --- | --- |
| `상품명` | 현재 매핑 시스템이 직접 먹는 제목 컬럼 |
| `외부콘텐츠ID` | 작품코드/상품ID/ISBN/공급자코드 등 매칭 보조키 |
| `판매금액` | S2 전송자료의 판매금액 후보 |
| `정산기준액` | S2 전송자료의 정산기준액 후보 |
| `상계금액` | S2 전송자료의 상계금액 후보. 없으면 0 후보 |

주의: 이 표는 "원형 헤더를 어떤 표준 필드로 볼지"의 1차 통일표다. 금액의 세전/세후/수수료 차감 의미는 adapter 구현 전에 플랫폼별로 추가 검증해야 한다.

- 병합 셀 템플릿 존재
- 헤더 시그니처 안정적

판단:

- adapter 우선순위 높음.
- 수수료를 S2 `상계금액`으로 넣을지 정책 확인 필요.

## 인간 수정 가능성 vs 원형 변형 신호

이번 배치에서 파일명 기반으로 인간 가공 가능성이 보이는 단서는 다음이다.

| 플랫폼 | 파일명 단서 | 판단 |
| --- | --- | --- |
| 네이버 | `(통합)` | 선투자/통합 원형일 수도 있고 사람이 합친 파일일 수도 있음. 별도 adapter 또는 제외 후보 |
| 네이버 | `_2차` | 같은 월 추가 정산/재정산 가능성. 원형 변형으로 분리 |
| 네이버 | `복사본` | 인간 복사 가능성 높음. adapter fixture 제외 후보 |
| 미소설 | `_확장체크` | 인간 확인/검수 목적 가능성. fixture 제외 후보 |
| 미스터블루 | `_프모비교` | 인간 비교/가공 목적 가능성. fixture 제외 후보 |

반대로 아래는 사람 수정이라기보다 플랫폼 원형 export 차이로 우선 보는 게 맞다.

| 플랫폼 | 변형 |
| --- | --- |
| 조아라 | 장르/단행본/작품코드 포함 여부에 따른 원형 차이 |
| 스토린랩 | 원스/북패스/정액권/B2BC 계열 차이 |
| 판무림 | 세부내역 vs 집계/요약 시트 차이 |
| 구글 | transaction report 원형 |
| 카카오 | 공급가액/세액/공급대가가 분리된 report 원형 |

## 1/3차 결론

바로 구현 가능한 후보:

```text
네이버 일부
조아라
카카오
큐툰
노벨피아
블라이스
```

adapter는 가능하지만 세부 분기가 필요한 후보:

```text
스토린랩
판무림
미스터블루
미소설
```

지금 바로 자동화하면 위험한 후보:

```text
윌라
한아름
구글
```

1/3차만 봐도 결론은 명확하다. 플랫폼별 초기 원형을 표준화할 수는 있지만, `플랫폼 폴더명 -> 단일 헤더 치환표`가 아니라 `플랫폼 + 헤더 시그니처 + 시트명` 기준 adapter로 가야 한다.
