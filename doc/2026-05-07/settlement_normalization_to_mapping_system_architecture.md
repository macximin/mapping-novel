# 정산상세 표준화 레이어 -> 매핑 시스템 투입 구조

- 작성일: 2026-05-07
- 목적: 플랫폼별 `정산상세` 원형 파일을 현재 매핑 시스템이 먹을 수 있는 통일 스키마로 변환하는 구조를 정의한다.
- 전제: `정산상세` 원형은 플랫폼별로 스키마가 다르며, 단순 `read_excel(header=0)` 또는 단일 헤더 치환표로는 안정 처리할 수 없다.

## 한 줄 결론

가능하다. 단, `정산상세` 원형을 매핑 시스템에 직접 먹이는 것이 아니라, 앞단에 `SettlementSchemaAdapter` 기반 표준화 레이어를 둬야 한다.

```text
플랫폼별 정산상세 원형
-> 헤더/시트/플랫폼 감지
-> SettlementSchemaAdapter
-> 표준 정산서 DataFrame
-> 현재 매핑 시스템 build_mapping()
-> S2 전송자료 생성
```

## 현재 매핑 시스템이 먹는 것

현재 `mapping_core.build_mapping()`은 입력 3개를 받는다.

| 입력 | 현재 요구 스키마 |
| --- | --- |
| S2 콘텐츠 리스트 | 콘텐츠명 계열 컬럼 + 판매채널콘텐츠ID 계열 컬럼 |
| 플랫폼 정산서 | 상품명/작품명/제목 계열 컬럼 하나 |
| 키다리스튜디오 소설 마스터 | 콘텐츠명 + 콘텐츠ID |

현재 매핑 시스템에서 정산서는 사실상 제목 컬럼 하나만 사용한다.

정산서 제목 후보:

```text
컨텐츠
타이틀
작품명
도서명
작품 제목
상품명
이용상품명
상품 제목
ProductName
Title
제목
컨텐츠명
콘텐츠명
시리즈명
```

따라서 현재 시스템에 최소로 먹이려면 표준화 레이어가 `상품명` 컬럼만 만들어도 된다.

## 우리가 원하는 최종 입력 스키마

하지만 목표는 단순 매핑 결과가 아니라 S2 전송자료 생성이다. 그러려면 표준 정산서 DataFrame은 다음 필드를 가져야 한다.

| 표준 필드 | 설명 | 현재 매핑 필요 | S2 전송자료 필요 |
| --- | --- | --- | --- |
| `상품명` | 매칭용 제목. `raw_title`의 표준 alias | Y | 간접 필요 |
| `외부콘텐츠ID` | 원본 정산서의 작품코드/상품ID/ISBN 등 | N | 매칭 보조 |
| `판매금액` | S2 전송자료의 판매금액 | N | Y |
| `정산기준액` | S2 전송자료의 정산기준액 | N | Y |
| `상계금액` | S2 전송자료의 상계금액. 없으면 0 후보 | N | Y |
| `source_file` | 원본 파일명 | 검증용 | 검증용 |
| `source_sheet` | 원본 시트명 | 검증용 | 검증용 |
| `source_row_no` | 원본 행 번호 | 검증용 | 검증용 |
| `platform` | 판매처/플랫폼 | 검증용 | 검증용 |
| `schema_adapter` | 적용된 adapter 이름 | 검증용 | 검증용 |
| `schema_status` | `parsed`, `needs_manual_review`, `schema_error` | 검증용 | 검증용 |

즉 현재 매핑 시스템을 유지하면서도 미래 S2 전송자료까지 고려한 표준 스키마는 다음과 같다.

```text
상품명
외부콘텐츠ID
판매금액
정산기준액
상계금액
source_file
source_sheet
source_row_no
platform
schema_adapter
schema_status
```

## S2 전송자료 출력 스키마

최종 S2 전송자료는 아래 4컬럼으로 고정한다.

```text
판매채널콘텐츠ID
판매금액
정산기준액
상계금액
```

`판매채널콘텐츠ID`는 정산상세 원본에서 직접 오는 값이 아니다. 대부분의 원본 정산상세에는 S2의 `판매채널콘텐츠ID`가 없으므로, 표준화된 `상품명`/`외부콘텐츠ID`를 KISS/S2 판매채널콘텐츠 목록과 매칭해서 붙여야 한다.

## 왜 단일 헤더 치환표가 아닌가

정산상세 전수조사 결과:

| 항목 | 값 |
| --- | ---: |
| 정산상세 파일 | 318 |
| 시트 | 360 |
| redaction 후 헤더 시그니처 | 61 |
| 플랫폼별 스키마 그룹 | 80 |
| 병합 셀 포함 시트 | 175 |
| 헤더 후보 없음/수동 검토 | 66 |

헤더 행도 1행, 2행, 3행, 10행, 27행 등으로 갈라져 있다. 따라서 다음 방식은 위험하다.

```text
read_excel(header=0)
컬럼명 replace
바로 build_mapping()
```

이 방식은 헤더 위치가 다른 파일, 집계 시트, 병합 셀, 보조 시트, Google transaction 형식 같은 파일에서 깨진다.

## 표준화 레이어의 역할

표준화 레이어는 3단계를 수행한다.

### 1. 헤더 위치 찾기

파일마다 헤더 행이 다르다.

예:

| 케이스 | 처리 |
| --- | --- |
| 1행 헤더 | 바로 사용 |
| 2~3행 헤더 | 상단 설명행 skip |
| 10행 이후 헤더 | 플랫폼별 adapter 필요 |
| 병합 셀/다중 헤더 | 병합된 상위 헤더를 보존하거나 flatten |
| 헤더 후보 없음 | manual review 또는 고정 위치 adapter |

### 2. 헤더명 통일

대표 치환 방향:

| 원본 헤더 후보 | 표준 필드 |
| --- | --- |
| `작품명`, `상품명`, `컨텐츠`, `타이틀`, `제목`, `콘텐츠명`, `시리즈명` | `상품명` |
| `작품코드`, `상품ID`, `채널상품ID`, `ISBN`, `컨텐츠No`, `시리즈ID` | `외부콘텐츠ID` |
| `판매금액`, `판매액`, `총매출`, `총매출액`, `공급대가` | `판매금액` 후보 |
| `정산금액`, `정산액`, `정산총액`, `공급가액`, `입금액` | `정산기준액` 후보 |
| `상계금액`, `공제수수료`, `수수료`, `마켓수수료`, `취소금액` | `상계금액` 후보 |

### 3. 플랫폼별 의미 보정

가장 중요한 단계다.

같은 `정산금액`이라도 플랫폼마다 의미가 다를 수 있다.

예:

| 플랫폼 유형 | 주의점 |
| --- | --- |
| 네이버 | 쿠키/대여/소장/합계 구조. `합계`가 판매금액인지 정산기준인지 확인 필요 |
| 카카오 | `공급가액`, `세액`, `공급대가` 분리 |
| 구글 | transaction row 기반. 헤더 고정 위치/컬럼 의미 별도 adapter 필요 |
| 리디북스 | 대용량, 단권/세트/앱마켓 정산대상액 분리 |
| 하이북 | 구매/대여 판매금액과 정산금액 분리 |
| 토스/구루 | 면세/과세 정산금액 분리 |

따라서 표준화는 단순 rename이 아니라 플랫폼별 adapter가 해야 한다.

## SettlementSchemaAdapter 계약

각 adapter는 다음 책임을 가진다.

```text
detect(file_meta, workbook_meta, header_signature) -> confidence
parse(workbook, sheet) -> StandardSettlementRows
validate(rows) -> ValidationReport
```

### detect

파일이 자기 adapter 대상인지 판정한다.

판정 근거:

- 상위 폴더명
- 파일명
- 시트명
- 헤더 행 위치
- 헤더 시그니처
- 필수 컬럼 존재 여부

### parse

원본 시트를 표준 정산서 DataFrame으로 변환한다.

반드시 보존할 것:

- 원본 파일명
- 원본 시트명
- 원본 행 번호
- 원본 제목
- 원본 외부 ID 후보
- 원본 금액 후보

### validate

변환 결과가 S2 전송자료 후보로 쓸 수 있는지 검사한다.

검사 항목:

- 제목 없음
- 금액 없음
- 금액 숫자 파싱 실패
- 합계행/소계행/주석행 포함
- 음수/취소/환불 행
- 구매/대여/정액권 분리 필요
- 면세/과세 분리 필요
- `판매금액`, `정산기준액`, `상계금액` 의미 불확실

## 현재 매핑 시스템과의 연결

현재 `build_mapping()`은 정산서에서 제목 컬럼만 찾는다. 그래서 표준화 레이어가 아래처럼 컬럼을 만들어주면 현재 시스템에 바로 먹일 수 있다.

```text
상품명
판매금액
정산기준액
상계금액
source_file
source_sheet
source_row_no
platform
schema_adapter
schema_status
```

현재 시스템은 `상품명`만 사용하고, 나머지 컬럼은 결과 파일에 `정산서원본_...` 형태로 따라붙는다. 다음 단계에서는 `build_mapping()` 이후 `matched` 행만 골라 다음 출력 시트를 만들면 된다.

```text
S2전송자료:
판매채널콘텐츠ID = S2_판매채널콘텐츠ID
판매금액 = 정산서원본_판매금액
정산기준액 = 정산서원본_정산기준액
상계금액 = 정산서원본_상계금액
```

단, `S2_매칭상태 != matched`인 행은 전송자료에 넣으면 안 된다.

## 매칭 상태 정책

| 상태 | 처리 |
| --- | --- |
| `matched` | S2 전송자료 후보 |
| `no_match` | 전송자료 제외, 검토필요 |
| `ambiguous` | 전송자료 제외, 후보 목록 검토 |
| `blank_key` | 전송자료 제외, 원본 제목/행 검토 |
| `schema_error` | 전송자료 제외, adapter 규칙 검토 |
| `needs_manual_review` | 전송자료 제외, 플랫폼별 의미 확인 |

자동 확정 금지 원칙:

- 제목 정제키가 같은 후보가 여러 ID로 갈라지면 `ambiguous`
- 금액 의미가 불명확하면 `needs_manual_review`
- ID 컬럼에 제목 문자열 fallback 금지

## 구현 단계

1. 표준 스키마 dataclass/constant 정의
2. adapter registry 생성
3. 헤더 탐지기 구현
4. P0 adapter 구현
5. 표준화 결과 preview/export 구현
6. 현재 `build_mapping()`에 표준화된 DataFrame 투입
7. `S2전송자료` 시트 생성
8. manual review 리포트 강화
9. KISS/S2 읽기 API 자동 수집과 연결

## 최종 목표 흐름

```text
정산상세 원형 업로드
-> adapter 자동 감지
-> 표준화 preview
-> KISS/S2 후보 목록 확보
-> 매핑 실행
-> matched / no_match / ambiguous / schema_error 분리
-> matched만 S2전송자료 생성
```

이 구조가 잡히면 플랫폼별 원형이 달라도 매핑 시스템 입장에서는 항상 같은 음식을 먹게 된다.

```text
상품명
판매금액
정산기준액
상계금액
```

그리고 최종 산출은 항상 같다.

```text
판매채널콘텐츠ID
판매금액
정산기준액
상계금액
```
