- 작성일: 2026-05-07
- 대상 repo: `C:\Users\wjjo\Desktop\mapping-novel`
- 대상 공유 폴더: `\\172.16.10.120\소설사업부\판무팀_ssot\100_계산서_매출등록_자료`
- 레거시 루틴: `정산서` -> `매핑 시스템` -> `S2 전송자료`
- S2 전송자료 핵심 컬럼: `판매채널콘텐츠ID`, `판매금액`, `정산기준액`, `상계금액`
- KISS 구조 분석 issue: https://github.com/macximin/mapping-novel/issues/2

## 목적

현재 루틴의 핵심 병목은 "정산서 원형이 어떤 형태인지 확정되지 않은 상태에서 매핑 결과와 S2 전송자료를 만들고 있다"는 점이다. 따라서 개선은 KISS API 연결부터 시작하지 않고, 공유 폴더에 축적된 `정산상세` 원형 파일을 먼저 분석해 입력 스키마를 확정하는 순서로 진행한다.

최종 목표는 다음이다.

1. 업체/판매채널별 정산상세 파일 구조를 분류한다.
2. 정산상세 원본을 표준 내부 스키마로 변환한다.
3. KISS/S2 판매채널콘텐츠 목록을 읽기 API로 가져온다.
4. 표준 정산상세와 판매채널콘텐츠 목록을 매핑한다.
5. `판매채널콘텐츠ID`, `판매금액`, `정산기준액`, `상계금액` 형식의 S2 전송자료를 생성한다.
6. 미매핑/중복/애매한 후보를 자동 확정하지 않고 검토 리포트로 분리한다.

## 작업 원칙

- 공유 폴더는 조사 단계에서 읽기 전용으로만 접근한다.
- 원본 파일을 수정, 이동, 삭제, 이름변경하지 않는다.
- 원본 데이터 전체를 repo에 복사하지 않는다.
- 문서에는 로그인 ID, 비밀번호, Bearer 토큰, 실제 인증값을 기록하지 않는다.
- KISS 쓰기 API, 저장 API, 결재 API, 전표 API는 1차 범위에서 호출하지 않는다.
- KISS API는 판매채널콘텐츠 목록 조회 같은 읽기 API만 후보로 둔다.
- 매칭이 애매하면 자동 확정하지 않고 `ambiguous` 상태로 남긴다.

## 현재 파악된 전제

KISS `ext-sale-colct-reg` 화면은 Vue SPA이며, 내부 REST API를 호출한다. API base는 `https://kiss-api.kld.kr`이고 로그인 후 Bearer 토큰을 사용한다.

읽기 API 후보:

| API | 용도 |
| --- | --- |
| `GET /ssmgmt/cmm/schn` | 거래처 기준 판매채널 목록 조회 |
| `GET /sale/ext/ext-salm/schn-ctns` | 판매채널콘텐츠 검색 |
| `GET /sale/ext/ext-salm` | 외부매출현황 목록 조회 |
| `GET /sale/ext/ext-salm/{extSalmId}/sale-colct` | 외부매출 상세 콘텐츠 목록 조회 |

콘텐츠형태 코드:

| 코드 | 의미 |
| --- | --- |
| `101` | 웹툰 |
| `102` | 웹소설 |
| `301` | 영상 |

이번 개선 대상은 키다리스튜디오 소설이므로, 기본 후보는 `ctnsStleCd=102`다. 단 이 값만으로 키다리스튜디오 소설이 완전히 확정되는 것은 아니며, 거래처 코드와 판매채널 ID allowlist가 추가로 필요하다.

| 금액 컬럼 | 매출액/판매금액/정산금액/공제/상계 등 |
| 상품 식별 컬럼 | 상품명, 작품명, 콘텐츠명, 외부상품ID 등 |

- 각 파일군마다 필수 컬럼과 선택 컬럼이 기록된다.
- 스키마 불명 파일은 별도 `needs_manual_review` 목록으로 분리된다.

### 4단계: KISS/S2 판매채널콘텐츠 목록 수집 설계

목표:

- 사용자가 KISS 사이트에서 S2 콘텐츠 리스트를 수동 다운로드하지 않도록 만든다.
- 단, 1차에서는 읽기 API만 사용한다.

후보 흐름:

1. KISS 로그인 세션 생성
2. 허용된 거래처 코드/판매채널 ID 로드
3. `GET /ssmgmt/cmm/schn`으로 거래처의 판매채널 확인
4. `GET /sale/ext/ext-salm/schn-ctns`에 `ctnsStleCd=102`, `schnIds`, `ctnsNm` 조건 적용
5. 응답을 내부 판매채널콘텐츠 스키마로 저장

내부 판매채널콘텐츠 스키마 초안:

| 필드 | 의미 |
| --- | --- |
| `schnCtnsId` | 판매채널콘텐츠ID |
| `schnId` | 판매채널ID |
| `schnNm` | 판매채널명 |
| `ctnsId` | 콘텐츠마스터ID |
| `ctnsNm` | 콘텐츠명 |
| `ctnsStleCdNm` | 콘텐츠형태명 |

산출물:

- `doc/2026-05-07/kiss_s2_read_api_collection_plan.md`

완료 기준:

- 쓰기 API 없이 S2 매핑 후보 목록을 확보하는 방법이 확정된다.
- API 실패 시 수동 업로드 fallback 정책이 정해진다.

### 5단계: 매핑 및 S2 전송자료 생성 설계

매칭 상태:

| 상태 | 의미 |
| --- | --- |
| `matched` | 유일 후보로 확정 |
| `no_match` | 후보 없음 |
| `ambiguous` | 후보가 2개 이상이라 자동 확정 금지 |
| `blank_key` | 매칭 키 생성 불가 |
| `schema_error` | 원본 스키마 문제로 처리 불가 |

결과 workbook 구조:

| 시트 | 내용 |
| --- | --- |
| `요약` | 입력 파일 수, 행 수, 성공/실패/중복 통계 |
| `S2전송자료` | `판매채널콘텐츠ID`, `판매금액`, `정산기준액`, `상계금액` |
| `행별매핑결과` | 원본 행 기준 전체 매핑 결과 |
| `검토필요` | 미매핑/중복/스키마 오류 |
| `중복후보` | ambiguous 후보 목록 |
| `입력검증` | 파일/시트/컬럼 검증 결과 |

완료 기준:

- S2 전송자료에는 확정 매칭된 행만 들어간다.
- 미매핑/중복 후보는 전송자료에서 제외되고 검토 시트로 분리된다.
- ID 컬럼에는 제목 문자열 fallback을 넣지 않는다.
