## 요약

결론부터 말하면, 대량 문자열 치환보다 특수 규칙 우선 방식이 맞다.

S2/정산서/콘텐츠마스터 매칭은 구조적으로 100% 자동 매칭될 수 없다. S2에 없는 콘텐츠마스터 코드는 사람이 만들어야 한다. 다만 이미 만들어져 있는 기준과 잘 붙는지는 별도 문제이고, 이 부분은 개선 가능하다.

현재 가장 안전한 방향:

1. S2 기준에서 `[사용안함]`, `(사용안함)`, `[사용금지]`, `(사용금지)` 행을 먼저 제거한다.
2. S2 콘텐츠명에 대해 명시적 특수 규칙을 먼저 적용한다.
3. 그 결과를 기존 `clean_title()`에 넣는다.
4. 정산서와 콘텐츠마스터도 같은 정규화 정책을 쓰되, 적용된 규칙과 충돌 후보를 리포트에 남긴다.
5. 자동 확정이 불가능한 것은 `수동등록후보`로 분리한다.

## 현재 코드 상태

이미 일부는 들어가 있다.

- `cleaning_rules.py`의 `drop_disabled_rows()`는 사용안함/사용금지 행을 제거한다.
- `extract_master_work_title()`는 `<작품명>_작가_코드_코드_확정` 형태와 `작품명_작가_코드_코드_확정` 형태에서 작품명을 뽑는다.
- `kiss_payment_settlement.to_s2_lookup()`는 S2 원천을 lookup으로 만들 때 `extract_master_work_title()`을 적용한다.

하지만 아직 부족한 점:

- 이 규칙이 “특수 규칙”으로 명시적으로 감사되지 않는다.
- `build_mapping()` 단계에서는 S2 lookup이 이미 정리됐다고 가정한다.
- 수동 업로드한 `S2 기준 리스트`가 이미 정리되지 않았으면 동일한 특수 규칙이 보장되지 않는다.
- 어떤 규칙이 적용됐는지 결과 엑셀에 남지 않는다.
- legacy 앱의 `미매핑_콘텐츠마스터명` 같은 사람 작업용 출력 흐름이 아직 약하다.

## 적대적 감리 1: 과정제 False Positive

공격 가정: 서로 다른 작품인데 `clean_title()`이 같은 키로 접는다.

관찰:

- `2026년 4월 무툰 정산.xlsx`는 537행이다.
- 원본 작품명 고유값 537개가 현재 clean 후 469개로 줄어든다.
- clean key 충돌 그룹은 68개다.
- 상당수는 `[단행본]` 제거로 의도된 중복일 수 있다.
- 하지만 `제1부/제2부`, 숫자, 권/회 제거는 진짜 다른 상품을 합칠 수 있다.

개선:

- clean 충돌을 자동 확정하지 말고 `중복후보` 또는 `정제충돌` 시트로 분리한다.
- 판매채널 필터 이후에도 `정제키 -> 판매채널콘텐츠ID`가 2개 이상이면 자동 선택 금지.
- `19금`, `2부`, `제1부/제2부`, `24/7`, `SSS급`, `EX급`, `MLB` 같은 보호 토큰은 별도 정책으로 둔다.

## 적대적 감리 2: 참조 데이터 전처리 순서

공격 가정: 사용안함 행이나 S2 원문 접미사가 후보 인덱스에 먼저 들어가서 정상 행을 오염시킨다.

관찰:

- 현재 `build_mapping()`은 S2와 IPS에서 `drop_disabled_rows()`를 먼저 호출한다.
- S2 원천을 `to_s2_lookup()`로 변환하면 `extract_master_work_title()`이 적용된다.
- 하지만 `수동 S2 기준 리스트` 업로드는 이미 정리된 파일이라고 가정한다.

개선:

- S2 기준이 원천이든 lookup이든, 후보 인덱스 생성 직전에 한 번 더 `normalize_reference_title()`을 적용한다.
- 적용된 규칙명을 `입력검증` 또는 별도 `정제감사` 시트에 남긴다.
- `사용안함/사용금지 제거 -> 특수 규칙 -> generic clean_title` 순서를 코드로 고정한다.

권장 순서:

```text
raw frame
-> drop_disabled_rows()
-> apply_special_title_rules()
-> clean_title()
-> candidate index
```

## 적대적 감리 3: 수동 작업 흐름 붕괴

공격 가정: 자동 매칭은 개선됐지만, 실제 업무자가 미매핑을 어디서 어떻게 처리해야 하는지 모른다.

legacy 앱은 단순하지만 사람 작업용 흐름이 있었다.

- 정산서 업로드 파일명 stem에 `_매핑.xlsx`를 붙여 결과를 냈다.
- 결과에 `미매핑_콘텐츠마스터명` 같은 사람이 볼 열이 있었다.
- 예시 `2026년 4월 무툰 정산_매핑.xlsx`에는 `미매핑_콘텐츠마스터명` 19개가 있었다.

현재 앱은 더 안전하지만, 사람에게 넘기는 산출물은 더 명시적이어야 한다.

개선:

- 결과 엑셀에 `수동등록후보` 시트를 추가한다.
- 포함 컬럼:
  - `정산서_콘텐츠명`
  - `정제_상품명`
  - `판매채널명`
  - `S2_매칭상태`
  - `IPS_매칭상태`
  - `추천_콘텐츠마스터명`
  - `수동작업사유`
- S2에 없는 코드는 자동 생성하지 않는다.
- 이미 S2에는 있는데 콘텐츠마스터가 없는 경우와, S2 판매채널콘텐츠ID 자체가 없는 경우를 분리한다.

## Quality-First 문자열 클리닝 방안

대원칙:

- 대량 치환보다 작고 검증 가능한 특수 규칙을 우선한다.
- 특수 규칙은 데이터에서 검증된 경우에만 추가한다.
- 룰 적용 전후를 감사 가능하게 남긴다.
- 충돌이 생기면 자동 확정하지 않는다.

권장 구조:

```python
@dataclass(frozen=True)
class TitleRuleResult:
    original: str
    precleaned: str
    clean_key: str
    rule_name: str
    confidence: str
```

권장 함수:

```python
def normalize_reference_title(value: object, *, source: str) -> TitleRuleResult:
    raw = text(value)
    precleaned, rule_name = apply_special_title_rules(raw, source=source)
    clean_key = clean_title(precleaned)
    return TitleRuleResult(raw, precleaned, clean_key, rule_name, "high")
```

적용 위치:

- `kiss_payment_settlement.to_s2_lookup()`
- `mapping_core.build_mapping()`의 S2 후보 인덱스 생성 직전
- `mapping_core.build_mapping()`의 IPS 후보 인덱스 생성 직전
- 정산서 제목도 같은 결과 구조를 쓰되, 플랫폼 어댑터가 이미 잘라낸 제목은 과한 특수 규칙 적용을 피한다.

## 특수 규칙 v1

### S2 지급정산 원천 접미사 제거

목표:

```text
{작품명}_{작가명}_{바로북저작권코드}_{바로북상품코드}_확정
-> {작품명}
```

단, 실제 데이터에는 `미연결`, `신작예정`, `선인세없음`도 들어온다. 따라서 숫자만 허용하면 안 된다.

권장 패턴:

```python
STRUCTURED_S2_CONFIRMED_TITLE_PATTERN = re.compile(
    r"^(?P<title>.+)_(?P<author>[^_]+)_(?P<right_code>[^_]+)_(?P<product_code>[^_]+)_확정$"
)
```

품질 게이트:

- `title`이 비어 있으면 적용하지 않는다.
- 같은 `clean_key`에 서로 다른 `title`이 2개 이상이면 자동 확정하지 않고 충돌 리포트로 보낸다.
- `_확정`이 마지막 토큰이 아닐 경우 적용하지 않는다.
- 적용 행 수와 고유 title 수를 기록한다.

## 구현 순서 제안

1. `cleaning_rules.py`에 `TitleRuleResult`와 `normalize_reference_title()` 추가
2. S2 원천/lookup 모두 후보 인덱스 직전에 `normalize_reference_title(source="s2")` 사용
3. IPS는 `normalize_reference_title(source="ips")` 사용
4. 정산서는 `normalize_reference_title(source="settlement")`를 쓰되 플랫폼 어댑터 결과를 우선
5. `input_validation`에 다음 항목 추가
   - 특수 규칙 적용 행 수
   - 특수 규칙 고유 원문 수
   - 특수 규칙 clean key 충돌 수
   - 보호 토큰 충돌 수
6. 결과 엑셀에 `정제감사` 시트 추가
7. 결과 엑셀에 `수동등록후보` 시트 추가
8. 테스트 추가 후 샘플 3종으로 회귀 확인

## 테스트 계획

필수 테스트:

- `[사용안함]` 행은 특수 규칙 적용 전에 제거된다.
- `재벌가 차남은 먼치킨_말리브해적_1003258_472_확정`은 `재벌가차남은먼치킨`으로 매칭된다.
- `마법 배운 재벌집 늦둥이_서오_1005041_미연결_확정`도 매칭된다.
- `조선 양과자 제과점_하늘곰_신작예정_신작예정_확정`도 매칭된다.
- `19금 속으로`은 현재처럼 숫자를 제거할지, 보호할지 정책 결정을 요구하는 테스트로 둔다.
- `미스터 프레지던트 2부`는 `2부` 제거 여부를 정책 결정 테스트로 둔다.
