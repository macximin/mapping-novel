# 문자열 클리닝 및 정산서-S2 매핑 고도화 계획

작성일: 2026-05-09

## 핵심 판단

정산서-S2 매핑의 기준은 `지급정산관리`다.

이 계획의 실행 범위는 소설 파트다. 전체 S2에는 웹툰/기타/출판 데이터가 섞여 있으므로, 운영 액션 숫자는 `판매채널콘텐츠_콘텐츠형태 = 소설` 및 IPS 담당부서 기준 확정 소설 담당부서 데이터로 좁혀서 본다.

`판매채널콘텐츠`는 서비스 가능 채널을 보여주는 보조 자료이며, 여기에 `판매채널콘텐츠ID`가 있다고 해서 정산 전송에 사용할 수 있는 ID라고 확정하지 않는다.

`[정산정보없음]` suffix rename은 일부 케이스에서만 유효하다. 같은 콘텐츠ID 안에 정상 지급정산 채널과 정산정보 없는 채널이 섞인 경우, 콘텐츠명 단위 suffix는 정상 채널까지 차단하므로 버그를 만들 수 있다. 따라서 애매한 케이스는 rename하지 않고, 판매채널콘텐츠ID 단위 차단표와 검토 체크리스트로 관리한다.

## 운영 방침

1. 실제 S2 대조 결과를 우선한다.

정산서 샘플에 등장했는지 여부는 우선순위 보조 정보다. 핵심 기준은 `소설 판매채널콘텐츠에는 있으나 지급정산관리에는 없는가`다.

2. A/B급은 rename하지 않고 체크리스트로 관리한다.

정산서 등장 여부와 무관하게 확정 담당부서 A/B급 381개는 우선 판매채널콘텐츠ID 단위 체크리스트로 둔다. 콘텐츠명 suffix를 붙여 전체 콘텐츠를 막지 않는다.

3. C급은 기존처럼 선차단한다.

`[사용안함]`, `(사용안함)`, `[사용금지]`, `(사용금지)`는 매핑 기준에서 제외한다. 최신 대조 기준 C급은 3,337개다. 이미 앱의 `drop_disabled_rows()` 흐름에 반영되어 있다.

4. 정산정보없음은 문자열 클리닝 문제가 아니라 매핑 차단 문제로 분리한다.

문자열에 `[정산정보없음]`을 붙이거나 지우는 방식으로 해결하지 않는다. 실제 차단 기준은 판매채널콘텐츠ID 단위 목록이어야 한다.

5. 확정 담당부서 261개 혼합 케이스는 특별 처리한다.

같은 콘텐츠ID 안에 정상 정산 채널과 정산정보 없는 채널이 섞인 확정 담당부서 261개는 콘텐츠명 suffix로 처리하면 정상 채널을 같이 막는다. 개발팀 답변 전까지는 이 261개를 별도 차단/보강 후보로 유지한다.

## 문자열 클리닝 고도화 방향

현재 `clean_title()`은 범용 정규화가 강하다. 하지만 정산서-S2 매핑은 플랫폼별 표기 차이와 작품별 예외가 많아서 일괄 규칙만으로는 안전하지 않다.

앞으로는 비효율적으로 보이더라도 정성 규칙을 공식 구조로 받아들인다.

목표 파이프라인:

```text
raw title
-> source별 전처리
-> 구조형 제목 추출
-> 보호 토큰 처리
-> 일반 정규화
-> 특별 규칙 적용
-> 충돌/위험 감리
-> clean_key 생성
```

source 구분:

- `settlement`: 플랫폼 정산서 원본
- `s2_payment`: S2 지급정산관리 기준
- `s2_service`: S2 판매채널콘텐츠 보조자료
- `ips`: IPS 콘텐츠마스터

## 특별 규칙 테이블

특정 작품은 범용 정규화가 아니라 명시 규칙으로 통일한다.

예상 파일:

- `data/title_normalization_rules.csv`

초안 컬럼:

```csv
rule_id,enabled,source,scope_type,scope_value,match_type,raw_pattern,normalized_title,reason,confidence
```

규칙 종류:

- `exact`: 원문 완전 일치
- `contains`: 부분 포함
- `regex`: 정규식
- `channel_scoped`: 특정 판매채널에서만 적용
- `content_id_scoped`: 특정 콘텐츠ID에서만 적용
- `blocklist`: 매핑 제외

기본 원칙:

- 넓은 contains/regex는 최소화한다.
- 판매채널별 의미가 다른 제목은 channel scope를 둔다.
- 콘텐츠ID를 알 수 있는 경우 content_id scope를 우선한다.
- 규칙 적용 전후를 결과 엑셀에 남긴다.

## 보호 토큰 정책

다음 토큰은 무조건 제거하지 않는다.

- `19금`
- `2부`, `제1부`, `제2부`
- `24/7`
- `SSS급`
- `EX급`
- `MLB`
- 권/화/부/회 숫자

정책:

- 기본은 보존한다.
- 제거해도 충돌이 없다는 근거가 있을 때만 제거한다.
- 제거 후 같은 clean key에 복수 원문이 몰리면 자동 매핑하지 않는다.
- 반복 검증된 케이스만 특별 규칙으로 승격한다.

## 구현 순서

1. `TitleRuleResult` 구조를 추가한다.

필드 후보:

- `original`
- `source`
- `precleaned`
- `clean_key`
- `rules_applied`
- `risk_flags`

2. `normalize_title()` wrapper를 추가한다.

기존 `clean_title()`은 유지하되, 새 매핑 경로에서는 source/context를 받는 wrapper를 사용한다.

```python
normalize_title(value, *, source, sales_channel=None, content_id=None)
```

3. 특별 규칙 테이블을 읽는다.

초기에는 CSV로 시작한다. 규칙이 늘면 JSON/YAML로 옮길 수 있다.

4. 보호 토큰 감리를 추가한다.

보호 토큰이 감지되면 제거하지 않고 risk flag에 기록한다.

5. `클리닝감리`와 `수동등록후보` 시트를 결과 workbook에 추가한다.

6. 정산정보없음 차단표를 별도 입력으로 둔다.

예상 파일:

- `data/blocked_sales_channel_content_ids.csv`

초안 컬럼:

```csv
sales_channel_content_id,content_id,sales_channel_name,block_reason,grade,source_path,review_status
```

7. A급 275건과 rename 위험 351건을 회귀 테스트 fixture로 삼는다.

자동화는 이 케이스들을 오매핑하지 않는 것을 먼저 보장해야 한다.

## 완료 기준

1. `[정산정보없음]` suffix 없이도 판매채널콘텐츠ID 단위 차단이 가능하다.
2. 같은 콘텐츠ID 안에 정상 채널과 문제 채널이 섞여도 정상 채널은 차단하지 않는다.
3. 보호 토큰이 들어간 제목을 무리하게 접지 않는다.
4. clean key 충돌이 있으면 자동 확정하지 않는다.
5. 사람이 볼 수 있는 `클리닝감리`와 `수동등록후보`가 결과물에 포함된다.
6. 레거시의 `미매핑_콘텐츠마스터명` 역할이 새 workbook에서 복구된다.
7. 최신 S2 대조 기준 전체 4,710개/1,373개/351개와 확정 소설 담당부서 381개/261개를 재현 가능한 산출물로 다시 만들 수 있다.
