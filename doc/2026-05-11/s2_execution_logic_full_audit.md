# S2 실행 로직 전수조사

작성일: 2026-05-11

## 결론

현재 앱 기준의 사용 가능 S2 판매채널콘텐츠ID는 다음 순서로 확정한다.

1. `[5031] 정산정보(지급)관리`를 JSON API로 전량 조회한다.
2. `pageSize=1000000`, `ctnsStleCd=102`를 기본으로 쓰고 원천 XLSX를 만들지 않는다.
3. JSON에서 필요한 컬럼만 compact lookup으로 저장한다.
4. `[사용안함]`, `[사용금지]`, `[정산정보없음]`류 표식 행을 제외한다.
5. S2 `정산정보 누락 건`(`/stm/stm`)에 등재된 판매채널콘텐츠ID를 제외한다.
6. 플랫폼별 정산서 파일명 또는 수동 선택으로 실제 S2 판매채널명을 확정한다.
7. 해당 판매채널로 S2 후보를 좁힌 뒤 제목 정제키로 매칭한다.
8. 매칭 불가능 행은 리포트로 분리하고, 청구정산 후보 여부를 보조 라벨로 붙인다.
9. S2 전송자료는 `matched`, 후보 1개, 분리사유 없음, 금액 검증 통과 행만 생성한다.

사람이 파일을 열었거나 저장했다는 사실만으로 차단하지 않는다.
차단/경고 기준은 시트명, 헤더 위치, 헤더명, 반복 구간 구조, 금액 후보 컬럼 의미가 어댑터 계약과 맞는지다.

## 구현 상태

### `[5031] 지급정산관리` JSON 최신화

- 앱 실행 경로: `app.py`의 `run_s2_refresh`
- refresh script: `scripts/refresh_kiss_payment_settlement.py`
- API: `/mst/setl/pymt-setl`
- 기본 옵션: `--lookup-only --page-size 1000000 --content-style-code 102`
- cache: `data/kiss_payment_settlement_s2_lookup.csv`
- 2026-05-11 실행 결과:
  - API fetch: 145,061건
  - source rows: 141,693건
  - compact lookup: 124,717건
  - 판매채널콘텐츠ID 중복: 0건
  - disabled marker 잔존: 0건

### S2 `정산정보 누락 건` guard

- refresh script: `scripts/refresh_s2_reference_guards.py`
- API: `/stm/stm`
- cache: `data/s2_payment_missing_lookup.csv`
- 2026-05-11 실행 결과: 2,327건
- 앱 통합: S2 기준 로딩 직후 `판매채널콘텐츠ID` 기준 제외
- 현재 `[5031]` compact lookup과 ID overlap: 0건

의미:

- `[5031]` 지급정산관리 기준에는 보통 누락 건이 이미 없으므로 제외 수가 0일 수 있다.
- 그래도 수동 S2 업로드나 향후 S2 기준 흔들림이 들어오면 ID 단위로 차단한다.
- no-match 행과 `판매채널명 + 정제 제목`이 맞으면 `S2 정산정보 누락 건 등재` 사유를 붙인다.

### `[5467] 청구정산` guard

- refresh script: `scripts/refresh_s2_reference_guards.py`
- API: `/mst/setl/req-setl`
- cache: `data/s2_billing_settlement_lookup.csv`
- 2026-05-11 실행 결과: 868건
- 앱 통합: 지급정산 no-match 행에 `판매채널명 + 대표콘텐츠명 정제키` 기준 보조 라벨링

주의:

- 목록 API 응답에는 판매채널콘텐츠ID가 없다.
- 따라서 현재 청구정산 guard는 ID 확정 기준이 아니라 `청구정산 후보` 보조 증거다.
- 지급정산 matched 행은 제목이 같아도 청구 후보로 재분류하지 않는다.

### 앱 리포트 변화

매핑 결과에 다음 컬럼이 추가된다.

- `S2_정산정보누락_후보수`
- `S2_정산정보누락_판매채널콘텐츠ID목록`
- `S2_정산정보누락_콘텐츠ID목록`
- `청구정산_후보수`
- `청구정산마스터ID목록`
- `청구정산_계약ID목록`
- `S2_분리사유`

배치 요약에는 `누락 후보`, `청구 후보`, `누락 제외`가 표시된다.

### S2 전송자료 차단

구현 완료.

- `S2_매칭상태 != matched`이면 차단
- `S2_후보수 > 1`이면 차단
- `S2_분리사유`가 있으면 차단
- 금액 정책 미확정이면 차단
- 판매금액/정산기준액/상계금액 후보가 숫자가 아니면 차단

운영 의미:

- 매핑 리포트에서는 최신 등록일자 후보를 보여줘 작업 속도를 유지한다.
- 실제 S2 전송자료는 중복 후보와 청구/누락 분리 가능성이 있는 행을 막아 안정성을 우선한다.

## 적대적 감리 1회차: 기준 데이터 공격

공격:
`[5031]`만 보면 지급정산 없음의 원인이 누락인지, 청구정산인지, 계약 미등록인지 구분되지 않는다.

방어:
`/stm/stm` 누락 guard와 `/mst/setl/req-setl` 청구 guard를 별도 cache로 붙였다.

잔여 위험:
청구정산 목록 API에는 판매채널콘텐츠ID가 없어서, 청구 후보는 제목+채널 보조 판정이다.

## 적대적 감리 2회차: 수동 파일 공격

공격:
사용자가 오래된 S2 엑셀이나 수동 리스트를 올리면 JSON 최신화 기준을 우회할 수 있다.

방어:
수동 업로드에도 missing guard와 billing annotation을 적용하고, 화면에 예외 모드 경고를 띄운다.

잔여 위험:
수동 업로드는 `[5031]` 최신 JSON 자체를 보장하지 못한다. 운영 기본값은 로컬 S2 기준 사용이다.

## 적대적 감리 2.5회차: 사람 손 흔적 공격

공격:
작성자/최종수정자, 저장 시각 차이 같은 메타데이터만 보고 파일을 오염으로 판정하면 실제로 파싱 가능한 정산서를 버릴 수 있다.

방어:
`사람가공` 같은 파일명만으로 차단하지 않는다. 사람이 손댔더라도 필수 헤더와 행 구조가 살아 있으면 매핑 입력으로 사용한다.

차단/경고 기준:

- 시트명이 어댑터 scope에서 빠짐
- 헤더명을 찾지 못함
- 헤더 위치 또는 반복 구간 구조가 깨짐
- 상품명/금액 후보 컬럼이 비거나 의미가 확정되지 않음
- 파일 자체 XML이 깨져 workbook을 열 수 없음

## 적대적 감리 3회차: 자동 매칭 공격

공격:
같은 정제키에 후보 ID가 여러 개이면 최신 등록일자 자동 선택이 잘못될 수 있다.

현재 정책:
매핑 리포트에서는 중복 후보를 `ambiguous`로 차단하지 않고 최신 등록일자 후보를 자동 선택한다. 후보 목록은 `중복후보` 시트에 남긴다.

방어:
S2 전송자료 생성 단계에서는 `S2_후보수 > 1`을 차단한다.
