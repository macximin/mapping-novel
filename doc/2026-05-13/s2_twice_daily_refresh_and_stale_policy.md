# S2 2회 자동 업데이트 및 Stale 방지 정책

작성일: 2026-05-13

## 1. 운영 기준 변경

S2 자동 업데이트 기준을 1일 1회에서 1일 2회로 변경한다.

- 실행 시각: 매일 12:00, 24:00 KST
- stale 판정 기준: 마지막 성공 업데이트가 12시간을 넘기면 `확인 필요`
- 앱 표시 문구: `매일 12:00, 24:00 KST 정규 업데이트`

실제 스케줄러는 이 repo의 앱 코드 반영 후, 자동 업데이트를 실행하는 환경에서 별도로 변경한다.

## 2. 기대 효과

기존 10시 1회 업데이트는 오후 작업자가 오래된 오전 기준을 계속 사용할 수 있었다.

12시/24시 2회 업데이트로 바꾸면 stale window가 대략 절반으로 줄어든다.

예상 효과:

- 오전 작업: 24시 기준 또는 12시 직전 기준 사용
- 오후 작업: 12시 기준 사용
- 다음날 오전 작업: 24시 기준 사용

따라서 “전량 처리했는데 너무 오래된 S2 기준을 봤다”는 위험은 크게 줄어든다.

## 3. 그래도 남는 위험

2회 업데이트만으로 stale/silent failure가 완전히 사라지지는 않는다.

남는 위험:

1. 12시 또는 24시 업데이트 자체가 실패한다.
2. S2 본 기준은 갱신됐지만 guard 또는 콘텐츠 lookup만 실패한다.
3. summary 파일은 최신처럼 보이지만 실제 CSV가 이전 파일이다.
4. API가 정상 응답을 줬지만 row 수가 비정상적으로 적다.
5. 사용자가 갱신 완료 직전에 실행한다.

따라서 앱은 “오늘 날짜인지”가 아니라 “마지막 성공 업데이트가 12시간 이내인지”를 기준으로 표시한다.

## 4. 앱 반영 내용

앱 사이드바의 S2 기준 상태는 다음 기준을 사용한다.

- `data/kiss_payment_settlement_s2_lookup.csv` 행 수가 0이면 `확인 필요`
- 마지막 성공 업데이트 시각이 12시간 이내면 `사용 가능`
- 12시간을 넘기면 `확인 필요`
- summary의 `s2_lookup_rows`와 실제 CSV 행 수가 다르면 경고
- guard summary가 없거나 12시간 기준을 넘기면 경고
- 콘텐츠 lookup summary가 없거나 12시간 기준을 넘기면 경고
- 콘텐츠 lookup의 `fetch_failures` 또는 `target_missing_channels`가 1건 이상이면 경고

이 경고가 떠도 앱 실행 자체를 막지는 않는다. 대신 사용자는 관리자 최신화 요청 또는 수동 S2 업로드를 선택할 수 있다.

## 5. 스케줄러 변경 시 주의

GitHub Actions 또는 외부 스케줄러가 UTC 기준이면 KST 변환을 명확히 해야 한다.

KST 기준 실행 시각:

| KST | UTC cron |
| --- | --- |
| 12:00 | 03:00 |
| 24:00 | 15:00 |

주의할 점:

- 24:00 KST는 다음날 00:00 KST다.
- UTC 15:00에 실행하면 runner의 `date.today()`는 아직 전날 UTC 날짜일 수 있다.
- 가능하면 스케줄러에서 `TZ=Asia/Seoul`을 설정하거나, refresh script에 KST 기준 `--today` 값을 넘긴다.

## 6. 성공 기준

스케줄러 변경 후 아래를 확인한다.

1. 12시 실행 후 Streamlit 사이드바의 S2 기준 시간이 KST로 표시된다.
2. 24시 실행 후 다음날 기준으로 `사용 가능`이 표시된다.
3. summary 행 수와 실제 S2 CSV 행 수가 일치한다.
4. guard/콘텐츠 lookup 경고가 뜨지 않는다.
5. 콘텐츠 lookup summary의 `fetch_failures`가 0이다.

## 7. 최종 판단

12시/24시 2회 업데이트는 stale 위험을 실질적으로 낮춘다.

다만 “문제 없음”으로 운영하려면 2회 실행뿐 아니라 성공 판정도 같이 봐야 한다. 이번 앱 반영은 그 기준을 사이드바에 드러내는 쪽이다.
