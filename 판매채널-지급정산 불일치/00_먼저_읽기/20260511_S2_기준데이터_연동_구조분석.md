# S2 기준데이터 연동 구조분석

작성일: 2026-05-11

## 결론

`[5023] 판매채널 콘텐츠 조회`, `[5031] 정산정보(지급)관리`, `[5467] 정산정보(청구)관리`는 모두 화면 뒤에서 JSON API를 사용한다.

따라서 "기술적으로 전체 기준 데이터를 줄 수 없는 시스템"으로 보기는 어렵다. 쟁점은 기술 가능성보다 권한, 내부통제, 운영 정책, 공식 지원 범위다.

## 연동 옵션별 현실성

### 1. 조회 전용 API 제공

현실성: 높음

이미 화면이 사용하는 API가 존재한다. 개발팀이 제공해야 할 것은 신규 개발보다 다음에 가깝다.

- 호출 가능한 API와 파라미터 공식화
- 조회 전용 계정 또는 토큰 발급
- 허용 page size와 호출 주기 합의
- 응답 스키마 변경 시 공지
- 필요한 경우 `[5031]`용 서버 전체 다운로드 API 추가

권장 요구:

- 개인 계정 세션을 쓰지 않는 service account
- read-only 권한
- 대상 API allowlist
- 일 1회 또는 수동 최신화 정도의 낮은 호출 빈도

### 2. 조회 전용 DB view 또는 replica 접근

현실성: 중간

기술적으로는 가능하겠지만 보안/내부통제/DB 운영 부담 때문에 개발팀이 방어적으로 나올 가능성이 높다.

그래도 "최소 컬럼 view" 형태라면 요청 근거는 있다.

필요 view 예:

- `판매채널콘텐츠ID`
- `콘텐츠ID`
- `콘텐츠명`
- `판매채널ID`
- `판매채널명`
- `계약ID`
- `지급정산마스터ID`
- `지급정산상세ID`
- `청구정산마스터ID`
- `승인상태`
- `운영상태`
- `최종수정일시`

협상 포지션:

- API 공식화가 어렵다면 DB view를 대안으로 요청한다.
- 쓰기 권한 없는 계정, 특정 view만 접근, 접속 로그 보존을 조건으로 제시한다.

### 3. 일별 자동 스냅샷 파일 제공

현실성: 높음

개발팀 입장에서는 API 공개나 DB 접근보다 부담이 낮을 수 있다.

가능한 방식:

- 매일 새벽 S2 기준 데이터를 CSV/XLSX/Parquet로 생성
- 사내 공유 경로 또는 S3/스토리지에 저장
- 파일명에 기준일 포함
- 스키마 변경 시 공지

장점:

- 개인 계정 자동 로그인 불필요
- 운영/감사 로그 남기기 쉬움
- 매핑 시스템은 파일만 읽으면 됨

단점:

- 실시간성은 떨어진다.
- 장애 시 파일 생성 실패를 감지하는 절차가 필요하다.

### 4. 화면 엑셀 다운로드 유지

현실성: 가능하지만 최하위

현재 구조에서는 메뉴별 다운로드 방식이 일관되지 않다.

- `[5023]`: client-side grid export
- `[5031]`: client-side grid export
- `[5467]`: client-side grid export + server-side 전체 다운로드

특히 `[5031]`은 전체 145,061건이 API로 조회되는데 화면 다운로드 파일이 1,000건으로 제한되는 정황이 있다. 기준 데이터로 쓰기에는 위험하다.

## 개발팀 협의용 핵심 주장

1. 세 메뉴 모두 이미 화면 뒤에 read API가 있다.
2. `[5031]`은 API total이 145,061건이므로 백엔드 데이터는 전체 조회 가능하다.
3. 문제는 화면 엑셀 다운로드가 기준 데이터 수집 방식으로 부적합하다는 점이다.
4. `[5467]`에는 이미 서버 전체 다운로드 API가 있으므로 `[5031]`에도 유사 방식이 가능해 보인다.
5. 최선은 조회 전용 API 공식화이고, 어렵다면 일별 자동 스냅샷이 현실적 대안이다.

## 요청 우선순위

1. `[5023]`, `[5031]`, `[5467]` 조회 전용 API 공식 사용 승인
2. 매핑용 통합 read-only API 신설
3. 일별 자동 스냅샷 제공
4. `[5031]` 서버 전체 다운로드 API 추가
5. 최후 수단으로 화면 엑셀 다운로드 유지
