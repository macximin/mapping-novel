# 2026-05-11 청구정산 반영 계획 및 적대적 감리

## 1. 개발팀 답변으로 확정된 운영 기준

- 특정 판매채널콘텐츠ID만 단독 비활성화하는 기능은 없다.
- 삭제/비활성화 성격의 처리는 백오피스개발팀 작업요청 대상이며, 기등록 매출등록/선급비용/선수수익 검토가 필요하다.
- 기존 판매채널콘텐츠ID에 지급정산정보를 사후 생성하려면 해당 판매채널+콘텐츠 계약 등록이 필요하다.
- 수정은 IPS 계약변경 및 S2 단건 수정으로 처리하되, S2 단건 수정은 재무부 권한이다.
- 지급정산정보 누락 건은 S2 `정산정보 누락 건` 메뉴의 유통 탭에서 확인 가능하고, 개발팀 별도 추출도 가능하다.
- 동일 콘텐츠ID 안에 정상 채널과 정산정보 없는 채널이 섞이면 콘텐츠명 단위 차단이 아니라 판매채널콘텐츠ID 단위 보강/처리가 필요하다.

## 2. 2026-05-11 추가 확인

- `플랫폼 매출정산 처리 현황_신청2026-05-11.xlsx`: 782건, 이 중 `[정산정보없음]` 표식 560건.
- `청구정산_목록_20260511.xlsx`: 50건.
- `콘텐츠+목록_20260511.xlsx`: 29,827건.
- `[5031]정산정보(지급)관리_20260511.xlsx`: 1,000건. 이 파일은 스타일 메타데이터 문제로 일반 openpyxl 로딩이 실패하므로 내부 XML 또는 재저장본으로 읽어야 한다.
- 청구성으로 본 `카카오페이지(선투자)` 27건은 IPS live에서 `[정산정보없음]_` prefix를 롤백했다.
  - write 결과: `updated=26`, `already_named=1`, 실패 0.
  - verify 결과: `already_named=27`.
  - 정확히 `청구정산_목록`의 판매채널ID+콘텐츠명까지 일치한 건은 1건이고, 나머지 26건은 청구성 채널 기준 롤백이다.

## 3. 다음 실행 계획

1. 판정 기준을 `지급정산 없음 = 정산정보없음`에서 `지급정산 없음 -> 청구정산 여부 확인 -> 혼재 여부 확인 -> suffix/차단/보강 분기`로 바꾼다.
2. `청구정산/비지급정산 후보` 분류를 별도 추가한다.
   - 우선 후보: `카카오페이지(선투자)`, `카카오페이지(창작지원금)`, `네이버_장르(광고수익)`, `네이버_연재(광고수익)`, `리디북스(이벤트)`, `봄툰 (원작 정산)`.
   - 2026-05-11 파일 기준 아직 검토가 필요한 표식 후보: 창작지원금 10건, 광고수익 20건, 이벤트 17건, 원작 정산 7건.
3. 혼재 콘텐츠는 콘텐츠명 suffix 금지로 고정한다.
   - `지급정산관리_존재_채널수 > 0`이고 특정 판매채널콘텐츠ID만 누락이면 판매채널콘텐츠ID 단위 차단/보강 큐로 보낸다.
4. 산출물을 네 갈래로 재생성한다.
   - 지급정산 OK
   - 청구정산/비지급정산 후보
   - 판매채널콘텐츠ID 단위 보강 또는 개발팀 작업요청 대상
   - 콘텐츠명 suffix 차단 대상
5. 앱 로직에는 다음 보호장치를 넣는다.
   - 지급정산관리 기준에 없는 코드는 자동 매핑하지 않는다.
   - 청구정산 후보는 지급정산 누락 경고와 분리 표시한다.
   - 혼재 콘텐츠는 콘텐츠ID가 아니라 판매채널콘텐츠ID로만 차단한다.

## 4. 적대적 감리 1회차: 청구성 오분류 위험

공격: 채널명이 `선투자`, `창작지원금`, `광고수익`, `이벤트`, `원작 정산`이라고 해서 모두 청구정산이라고 볼 수는 없다. 지급정산관리에도 일부 유사 채널이 존재한다.

보정:

- 채널명 키워드는 확정이 아니라 `청구성 후보`로만 쓴다.
- 확정은 `청구정산 마스터ID`, `계약ID`, 판매채널ID+콘텐츠명 일치, 또는 S2 메뉴 확인으로 승격한다.
- 키워드만 맞는 건 롤백 또는 suffix 제외 전에 검토 큐를 남긴다.

## 5. 적대적 감리 2회차: 롤백 후 오매핑 위험

공격: `[정산정보없음]_` prefix를 제거하면 앱이 다시 정산서 작품명과 매칭해 지급정산 코드가 없는 콘텐츠를 통과시킬 수 있다.

보정:

- 롤백은 IPS 명칭 정상화일 뿐, 지급정산 매핑 허용이 아니다.
- 앱에서는 지급정산관리 기준 미존재 행을 계속 차단한다.
- 청구정산 후보는 `지급정산 누락`이 아니라 `청구정산 경로`로 표시하고, 지급정산 전송 대상에서 제외한다.

## 6. 적대적 감리 3회차: 원천 데이터 불완전 위험

공격: 2026-05-11의 `[5031]정산정보(지급)관리` 파일은 1,000행이고, 전체 지급정산관리 기준으로 보기 어렵다. 2026-05-08 산출물과 2026-05-11 S2 메뉴 추출물의 범위도 다를 수 있다.

보정:

- 전체 대조에는 완전한 지급정산관리 export 또는 기존 93,403건 기준을 사용한다.
- S2 `정산정보 누락 건` 메뉴 export는 별도 공식 누락 큐로 취급한다.
- 각 산출물에는 기준일, 메뉴명, 행 수, 파일명, 한계사항을 남긴다.

## 7. 감리 후 최종 결론

현재 가장 안전한 방향은 `[정산정보없음]` suffix를 줄이고, `정산 유형/정산 경로` 분기를 늘리는 것이다.

- 지급정산관리 존재: 매핑 허용.
- 청구정산/비지급정산 후보: 지급정산 누락으로 보지 말고 별도 경로로 표시.
- 혼재 콘텐츠: 콘텐츠명 suffix 금지, 판매채널콘텐츠ID 단위 처리.
- 순수 누락 콘텐츠: 정산서 등장 여부와 청구정산 예외 여부를 본 뒤 `[정산정보없음]` suffix 또는 작업요청으로 보낸다.
